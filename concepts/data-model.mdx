DeepSpace stores app data in **collections** â€” typed tables backed by SQLite inside a Durable Object. Each collection is declared in a schema, baked into your worker at deploy time, and exposed to the client through `useQuery` / `useMutations` hooks.

<Info>
**Three things to remember:**
- A collection is a typed table. You declare it in `src/schemas/`, register it in `src/schemas.ts`, and the worker compiles it at deploy time.
- Every record is an envelope: `{ recordId, data, createdBy, createdAt, updatedAt }`. Your fields live under `.data`.
- Mutations resolve **optimistically** â€” the local store updates before the server confirms.
</Info>

## Collections and records

A collection is a named table with typed columns. A record is one row, wrapped in an envelope that carries metadata.

```ts
type Envelope<T> = {
  recordId: string         // unique ID generated by the worker
  data: T                  // your user-defined fields
  createdBy: string        // userId of the creator
  createdAt: string        // ISO timestamp
  updatedAt: string        // ISO timestamp
}
```

Your own fields live under `.data`. When you query a `todos` collection:

```tsx
const { records } = useQuery<{ title: string; completed: boolean }>('todos')
records[0].data.title       // âś“ "Buy milk"
records[0].recordId         // âś“ "rec_abc123"
records[0].title            // âś— undefined â€” this is a common bug
```

<Warning>
The envelope shape catches developers off guard. Always reach for `r.data.<field>`, not `r.<field>`. Use `r.recordId` to pass into `put` / `remove`.
</Warning>

## Defining a schema

Schemas live under `src/schemas/`, with the full list exported from `src/schemas.ts`. Every schema has `name`, `columns`, and `permissions`:

```ts
// src/schemas/items-schema.ts
import type { CollectionSchema } from 'deepspace/worker'

export const itemsSchema: CollectionSchema = {
  name: 'items',
  columns: [
    { name: 'title', storage: 'text', interpretation: 'plain' },
    { name: 'status', storage: 'text', interpretation: { kind: 'select', options: ['draft', 'published'] } },
    { name: 'priority', storage: 'integer', interpretation: 'number' },
  ],
  visibilityField: { field: 'status', value: 'published' },
  permissions: {
    viewer: { read: 'published', create: false, update: false, delete: false },
    member: { read: true, create: true, update: 'own', delete: 'own' },
    admin:  { read: true, create: true, update: true, delete: true },
  },
}
```

Register it:

```ts
// src/schemas.ts
export const schemas = [usersSchema, settingsSchema, itemsSchema]
```

Schemas are **baked in at deploy time** â€” there is no runtime schema registry. Adding or changing a schema requires a redeploy.

## Column types

Every column has a `storage` type and an `interpretation`:

| `storage` | SQLite type | What it holds |
|---|---|---|
| `text` | TEXT | Strings, IDs, JSON-encoded blobs |
| `integer` | INTEGER | Numbers, booleans, timestamps |
| `real` | REAL | Floating-point numbers |
| `blob` | BLOB | Binary data (rare in app code) |

`interpretation` tells the SDK how to encode/decode values:

- `'plain'` â€” pass through unchanged
- `'number'` / `'boolean'` â€” type coercion
- `{ kind: 'select', options: [...] }` â€” constrained enum
- `{ kind: 'json' }` â€” serialized object or array (auto `JSON.stringify` on write, auto `JSON.parse` on read)
- `{ kind: 'date' }` â€” ISO timestamp stored as text

```ts
{ name: 'tags', storage: 'text', interpretation: { kind: 'json' } }
// On write: pass the array directly â€” mutations.create({ tags: ['a', 'b'] })
// On read:  record.data.tags is already an array â€” don't JSON.parse
```

## Hooks: `useQuery` and `useMutations`

The two hooks every app uses:

```tsx
import { useQuery, useMutations } from 'deepspace'

type Item = { title: string; status: 'draft' | 'published' }

function ItemList() {
  const { records, status } = useQuery<Item>('items', {
    where: { status: 'published' },
    orderBy: 'createdAt',
    orderDir: 'desc',
    limit: 50,
  })

  const { create, put, remove } = useMutations<Item>('items')

  // create(data: Item) â†’ Promise<string>  (the new recordId)
  // put(id, patch: Partial<Item>) â†’ Promise<void>  (merge into existing row)
  // remove(id) â†’ Promise<void>
}
```

The hooks subscribe to a WebSocket the moment they mount and stream updates in real time. When any user (including you) creates, updates, or deletes a record, every open client sees the change within milliseconds.

### Optimistic vs confirmed mutations

`create` / `put` / `remove` apply changes locally first, then sync to the server. They resolve as soon as the local store is updated â€” usually before the server confirms.

For workflows that must wait for the server (e.g., capturing a server-generated ID), use the `*Confirmed` variants:

```ts
const { createConfirmed } = useMutations<Item>('items')
const recordId = await createConfirmed({ title: 'New', status: 'draft' })
navigate(`/items/${recordId}`)
```

`createConfirmed` doesn't resolve until the server has acknowledged the write.

## Scopes â€” picking the room

Records live inside a Durable Object. The `roomId` (or "scope") chooses which DO.

```tsx
<RecordProvider>
  <RecordScope roomId={SCOPE_ID} schemas={schemas} appId={APP_NAME}>
    <App />
  </RecordScope>
</RecordProvider>
```

`SCOPE_ID` from `src/constants.ts` defaults to `app:${APP_NAME}` â€” your app's main RecordRoom. Each scope is an independent DO with its own data.

Nesting `<RecordScope>` lets you mount additional rooms â€” for example, a per-conversation DO:

```tsx
<RecordScope roomId={`conv:${convId}`} schemas={CONVERSATION_SCHEMAS}>
  <ChatThread />
</RecordScope>
```

## How writes flow

When you call `useMutations.create(...)`, the SDK runs through six steps:

1. **Optimistic local apply.** The new record is added to the in-memory store; React re-renders.
2. **WebSocket dispatch.** A typed message is sent to the `AppRecordRoom` Durable Object.
3. **RBAC check.** The DO verifies the caller's JWT against the collection's permissions.
4. **SQLite write.** The DO persists the record in its local SQLite database.
5. **Broadcast.** The DO sends the new record to every connected client whose RBAC allows read access.
6. **Reconcile.** Other clients apply the update; the originating client confirms its optimistic state.

If the server rejects the write â€” an RBAC denial or a schema validation failure â€” the optimistic update rolls back automatically.

## What about big data?

DeepSpace records are tuned for **operational data** â€” collections of hundreds to tens of thousands of small rows, queried by client filters and updated frequently. For larger or analytical workloads:

- **Files / blobs** â†’ use [R2 file storage](/guides/file-uploads).
- **Vector search** â†’ declare a [custom Vectorize binding](/guides/custom-bindings) and call it from your worker.
- **Analytics** â†’ declare a custom Analytics Engine binding, or use the auto-provisioned `USAGE_EVENTS` dataset with [`meterUsage`](/sdk-reference/worker/bindings).
- **External SQL** â†’ declare a custom D1 database (with `runMigrations`) or a Hyperdrive binding to your own Postgres.
