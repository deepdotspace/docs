A scaffolded app is a normal Vite + React project with a Cloudflare Worker entry point. This page explains what each file does and which ones you'll edit most often.

If you only read one section, make it [The Durable Object manifest](#the-durable-object-manifest) — it's the bridge between your worker and the SDK.

## Top-level files

| File | Purpose |
|---|---|
| `worker.ts` | Hono worker and Durable Object class declarations. Edit when adding custom routes or DO classes. |
| `wrangler.toml` | Cloudflare config. The `name` field is your deploy subdomain. Declare custom bindings here. |
| `index.html` | HTML shell. Set `<html data-theme="...">` to pick a theme, update `<title>` and favicon. |
| `vite.config.ts` | Vite + `@cloudflare/vite-plugin` config. You rarely need to edit this. |
| `package.json` | Dependencies. The scaffold includes `deepspace`, `react`, `vite`, and a few Tailwind/UI peers. |
| `.dev.vars` | Local secrets (gitignored). Auto-managed for SDK keys; user-added keys are uploaded as `secret_text` on deploy. |

## `src/` — application code

### Pages and providers

| File | Purpose |
|---|---|
| `src/pages/_app.tsx` | Root layout. Wraps the tree in `ToastProvider → DeepSpaceAuthProvider → AuthBoot → RecordProvider → RecordScope`. **Extend, don't replace.** |
| `src/pages/home.tsx` | Public landing route (`/`). |
| `src/pages/(protected)/_layout.tsx` | Applies `<AuthGate>` to every route inside the `(protected)/` folder. |
| `src/pages/(protected)/*.tsx` | Gated routes. Add a file here for a new authenticated page. |
| `src/components/Navigation.tsx` | Top navigation with auth-aware controls. Reads entries from `src/nav.ts`. |
| `src/components/ui/` | Shadcn-style primitives — `Button`, `Dialog`, `Toast`, `Card`, `EmptyState`, and more. |

Routing is file-based via [generouted](https://github.com/oedotme/generouted): every file under `src/pages/` becomes a route. Folder names in parentheses (`(protected)/`) are route groups — they apply layouts without showing up in the URL.

### Data layer

| File | Purpose |
|---|---|
| `src/schemas.ts` | Exports the array of every collection schema in the app. |
| `src/schemas/` | One file per collection (`users-schema.ts`, `admin-schema.ts`, your custom collections). |
| `src/actions/index.ts` | Server actions — privileged worker functions called via `POST /api/actions/:name`. |
| `src/constants.ts` | `APP_NAME`, `SCOPE_ID`, and role re-exports. |

Schemas are imported by `worker.ts` and baked into the bundle at deploy time. There's no runtime schema registry — adding or changing a schema requires a redeploy.

### Theming and navigation

| File | Purpose |
|---|---|
| `src/themes.ts` | Typed catalog of theme presets. |
| `src/themes.css` | Per-theme CSS variable blocks (15 presets ship with the scaffold). |
| `src/styles.css` | Tailwind v4 entrypoint with the slate baseline `@theme` block. |
| `src/nav.ts` | Top-nav entries. Add new pages here to make them appear in `Navigation.tsx`. |

### Optional surfaces

These exist only if your app needs them — install via `npx deepspace add <feature>` or create by hand.

| File | Purpose |
|---|---|
| `src/cron.ts` | Scheduled tasks for `AppCronRoom`. See [Scheduled jobs](/guides/scheduled-jobs). |
| `src/ai/tools.ts` | System prompt and tool allowlist for `/api/ai/chat`. |
| `src/ai/chat-routes.ts` | Hono handlers for the AI chat endpoints. |
| `src/integrations.ts` | Per-integration billing config (`developer` vs `user`). |
| `src/subscriptions.ts` | Subscription plan manifest for Stripe billing. |
| `src/products.ts` | One-time product manifest. |

## `tests/` — Playwright specs

| File | Purpose |
|---|---|
| `tests/smoke.spec.ts` | Page-load tests, navigation, auth UI presence. |
| `tests/api.spec.ts` | API route status codes, response shapes, auth gating. |
| `tests/collab.spec.ts` | Two-user real-time sync tests. |
| `tests/playwright.config.ts` | Playwright config with `baseURL` and `webServer`. |

Run with `npx deepspace test`. See [Testing](/guides/testing).

## The Durable Object manifest

`worker.ts` declares the set of DO classes the app uses in a single constant:

```ts
export const __DO_MANIFEST__ = {
  RECORD_ROOMS:   { class: 'AppRecordRoom',   sqlite: true },
  YJS_ROOMS:      { class: 'AppYjsRoom',      sqlite: true },
  CANVAS_ROOMS:   { class: 'AppCanvasRoom',   sqlite: true },
  PRESENCE_ROOMS: { class: 'AppPresenceRoom', sqlite: true },
  CRON_ROOMS:     { class: 'AppCronRoom',     sqlite: true },
} as const satisfies DOManifest
```

The `Env` interface derives its DO bindings from this constant:

```ts
interface Env extends DOBindings<typeof __DO_MANIFEST__> {
  // ...your secrets and custom bindings
}
```

The deploy pipeline reads `__DO_MANIFEST__` to provision the right DO bindings on Cloudflare. Don't remove classes you no longer use without clearing their stored data first — the records persist across deploys.

## Conventions

- **Extend `_app.tsx`, don't rewrite it.** Add schemas, pages, and nav entries; leave the provider stack alone.
- **Import UI primitives from `../components/ui`, not from `deepspace`.** The scaffold's local kit has its own React contexts; mixing the two produces `useToast must be used within ToastProvider` at runtime.
- **Pages live under `src/pages/`.** generouted scans only this directory. Putting a page in `src/features/<name>/` results in 404s even if a nav link exists.
- **Never commit `.dev.vars`.** The scaffold's `.gitignore` covers it. To rotate a secret, edit the file and redeploy.
